# ADR 013 — Migração de MFA TOTP para Código enviado por E-mail

**Data:** 2026-07-14
**Status:** Aprovada (decisão executada na sessão)

## Contexto e Motivação
A implementação anterior do MFA utilizava TOTP (Time-Based One-Time Password) via aplicativos como Google Authenticator e Microsoft Authenticator. 
Contudo, em ambientes locais de desenvolvimento e homologação, o uso do TOTP apresentou problemas críticos de usabilidade e integração devido a dessincronização de relógios:
- O relógio do computador (servidor local da API) comumente diverge em minutos (ou até anos, devido a testes com clock congelado/shifted) do relógio de celulares dos desenvolvedores e usuários homologadores (que sincronizam automaticamente via rede celular NTP padrão).
- Uma dessincronização maior que 60 segundos impossibilita a validação matemática do TOTP, gerando falhas falsas de autenticação ("Código MFA inválido").

Para solucionar esta barreira operacional mantendo a conformidade com as exigências de duplo fator de autenticação (MFA), optou-se por alterar a abordagem para autenticação multifator via e-mail.

## Decisão
1. **Pivô de Abordagem**: Substituir o validador TOTP por um fluxo de geração e envio de código temporário numérico de 6 dígitos via E-mail.
2. **Ciclo de Vida do Código**: 
   - O código possui comprimento de 6 dígitos.
   - O código possui tempo de expiração padrão de 5 minutos (configurado em banco).
   - O código é deletado/invalidado imediatamente após a validação inicial para prevenir Replay Attacks (Replay Guard).
3. **Resiliência de Envio (Fallback)**:
   - Utilizar a biblioteca Angus/Jakarta Mail `2.0.3` (residente no cache local) para enviar e-mails reais através de SMTP configurado.
   - Se o SMTP não estiver configurado ou offline, o sistema gera o e-mail, imprime um banner destacado em formato ASCII no console do Spring Boot contendo o código MFA para que os testadores possam ver o código de forma fluida sem travar o sistema.
4. **Cadastro de E-mail**: Registrar o e-mail de remetente e do administrador como `ronaldoguitarrista@gmail.com` para validar fluxos reais de correio eletrônico.

## Consequências
- Fim da dependência crítica de sincronização de relógio de hardware móvel (celular) com a API local.
- Processo de teste facilitado no terminal local.
- Envio real de códigos direto para o e-mail do testador/operador quando configurado no `application.properties`.
- Redução da complexidade de QR Code do frontend.
